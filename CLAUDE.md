# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

**End goal, stated explicitly because it governs prioritization:** enable Claude Code to do *agentic programming* of Desynced behaviors — read, review, and edit a real in-game behavior the way it would work with source code in a normal IDE, then write the result back into the live game. Everything else in this repo (the `.dcs` wire codec, the BSF pipeline, the Lua-backed interpreter) is **instrumental infrastructure in service of that goal**, not the goal itself.

Both halves of that goal have working answers:

- **Transport**: Desynced's own copy/paste (`Tool.GetClipboard`/`SetClipboard`, producing the `.dcs` strings the codec decodes) plus a VirtualBox host/guest shared clipboard gives a live, bidirectional read/write channel with zero game-side code. The `desynced-bsf` CLI (also runnable as `python -m blz.desynced_toolkit.bsf`) is built to sit in a shell pipeline against that clipboard.
- **Representation**: Desynced behaviors are genuine flow graphs (arbitrary-fan-in control edges, arbitrary-fan-out data edges — confirmed against real user behaviors, not just synthetic ones generated by this project's own tooling), so the adopted representation is BSF, a graph-native text format (`behavior_source_format.md`, implemented in `blz.desynced_toolkit.bsf`) — not tree-structured pseudocode, which was tried first (a Python-subset `AstCompiler`, removed 2026-07-14; git history only) and abandoned as a lossy fit.

**Repository split (2026-07-22):** the wire codec, BSF pipeline, and Lua-backed runtime/interpreter — the shareable "instrumental infrastructure" mentioned above — now live in a separate sibling repo, `../blz-desynced-toolkit/`, as the `blz-desynced-toolkit` package (importable as `blz.desynced_toolkit`). This repo depends on it via a `tool.uv.sources` local path dependency (see `pyproject.toml`) — the same sibling-directory convention already used for the game-data extract below. **This repo now holds only the me-specific half**: the deployed behavior library (`library/`), corpus-collection tooling (`scripts/`, `corpus/`), and personal behavior-design docs (`combat_squad_spec.md`, `hex_expansion_math.md`, `blight_magnifier_mining.md`, `observer_redesign.md`). Toolkit module docs, the wire/behavior-format gotchas index, and the toolkit's own test suite moved to that repo's own `CLAUDE.md` — read it before touching anything under `blz.desynced_toolkit`.

This repo is tooling, documentation, and design work built *on top of* an extract of the base game data assets for **Desynced** (by The Desynced Team), a factory/automation game with programmable robots — the Lua scripts, textures, sounds, and other assets that define game behavior. There is no build system for the game data itself — it's a read-only reference for analyzing and understanding game mechanics.

**The game data extract lives outside this repo.** You need your own local copy of it; this repo does not include or fetch it. **Convention:** place it as a sibling directory of this repo, named `desynced-game-data` (i.e. if this repo is at `/path/to/desynced/`, the extract goes at `/path/to/desynced-game-data/`) — everything in this repo that looks for it (tests, tooling) defaults to that location with no configuration. If you put it somewhere else, set the `DESYNCED_GAME_DATA` env var to its path. Every path mentioned in this doc as if it were local (`data/instructions.lua`, `ui/ui.lua`, `main.zip`, etc.) is **relative to that external directory**, not this repo's root. Both Bash and Read can reach it directly via absolute path without any special setup; use `/add-dir <path-to-your-extract>` if a given session needs it added explicitly. `blz.desynced_toolkit.assets.open_asset_source(...)` (in the sibling toolkit repo) takes that directory (or its `main.zip`) as a plain argument — nothing is hardcoded.

The entry point manifest is `def.json` (in the external directory), which defines all packages and their load order.

**Current/planned work lives in `todo.md`** (checked in, preferred over session-scoped task tools) — read it rather than reconstructing project state from this file or git history. `todo.md` holds only open items; resolved ones move to `history.md` (same topical section headers, not date/release-ordered) — check there for the design rationale and in-game findings behind past decisions.

## Package Architecture

The game uses a package system. Each package has an `entry` Lua file that sets `package.includes` (a list of files to load in order) and lifecycle callbacks:

- `package:init()` — runs at startup, registers data definitions into global tables
- `package:init_ui()` — UI-only initialization
- `package:setup_scenario(settings)` — called before scenario starts
- `package:validate_data()` — dev-mode data integrity checks
- `package:on_update(ver)` — migration logic for saved games upgrading from old versions

The packages defined in `def.json`:

| Package | Entry | Type | Purpose |
|---|---|---|---|
| `Data` | `data/data.lua` | Data | All game definitions (items, frames, components, etc.) |
| `UI` | `ui/ui.lua` | UI | In-game HUD, menus, input bindings |
| `FrontEnd` | `frontend/frontend.lua` | Scenario | Main menu and lobby |
| `Freeplay` | `scenarios/freeplay.lua` | Scenario | Primary open-world game mode |
| `Versus` | `scenarios/versus.lua` | Scenario | PvP scenario |
| `Resim/Registers1/Registers2/Signals/Nomad/TowerD` | `scenarios/*.lua` | Challenge | Puzzle/challenge scenarios |

## Core Data Definitions (`data/`)

All game definitions are registered into global `data.*` tables. Each definition file documents its own schema at the top as a comment block.

**Key definition tables:**

- `data.frames` — Entity "frames" (the physical body of a unit or building). Built with `Frame:RegisterFrame(id, def)`. Frame definitions include movement speed (non-zero = unit/bot), health, slots, initial components, and lifecycle callbacks (`on_placed`, `on_destroy`, `on_interact`).
- `data.components` — Attachable components that go into frame sockets. Define behavior, production recipes, power consumption, registers, and event callbacks (`on_add`, `on_remove`, `on_update`, `on_trigger`, `on_take_damage`).
- `data.items` — Inventory items with stack sizes and recipes.
- `data.instructions` — The visual programming instruction set. Each instruction has `func(comp, state, cause, ...)` implementing its behavior. Instructions are categorized: Flow, Unit, Global, Math, Move, Component, AutoBase.
- `data.behaviors` — Pre-encoded behavior programs (stored as base62-encoded strings).
- `data.techs` — Technology tree nodes with uplink recipes and unlock lists.
- `data.explorables` — Procedurally placed world events/encounters. Each explorable implements `GetRelevancy(x, y, info)` and `SpawnExplorable(x, y)`.
- `data.values` — Enum-like named values (colors, radar filter types, alien signals).
- `data.visuals` / `data.visualassets` — Visual representations and socket positions.
- `data.fx` — Sound/particle effects.
- `data.biomes`, `data.landfeatures`, `data.cliffs` — World generation parameters.
- `data.puzzles` — Minigame puzzle definitions.

**Recipe helpers** (defined in `data/utilities.lua`):
- `CreateProductionRecipe(ingredients, producers, amount)` — manufactured items/components
- `CreateMiningRecipe(miners)` — mined resources
- `CreateUplinkRecipe(ingredients, ticks)` — research via uplink
- `CreateConstructionRecipe(ingredients, ticks)` — placed buildings

## Faction & Entity System

The game world is divided into **factions**: `player` (human-controlled), `world` (neutral explorables), `bugs`, `alien`, `anomaly`. Trust relationships between factions determine hostility. Faction-level data persists in `faction.extra_data`.

**Entity registers** are the core of the programming system. Frames have a fixed number of registers (FRAMEREG_GOTO, FRAMEREG_STORE, etc.) that hold typed values (entities, items, numbers, signals). The visual behavior editor programs these by constructing instruction sequences.

**Behaviors** are programs attached to components (particularly `c_behavior` and `c_autobase`). The `data/library.lua` handles compiling encoded behavior strings into bytecode (`GetFactionBehaviorAsm`) and caching them.

## Action System

The game uses typed action dispatchers:

- `PlayerAction.*` — actions sent by a player, validated server-side
- `EntityAction.*` — actions applied to a specific entity
- `FactionAction.*` — actions applied to a faction
- `MapMsg.*` — broadcast map-wide messages
- `UIMsg.*` — UI-only callbacks (run only on local client)
- `Chat.*` — multiplayer chat message handlers
- `Delay.*` — deferred map-tick callbacks
- `MapRun.*` — simulation-side broadcast

## Explorable System

Explorables are the world's discoverable encounters. The file `data/explorables.lua` contains `Explorable_SpawnAt(x, y, discoverer, filter)` which selects an explorable by weighted `GetRelevancy` scores based on tile data (blightness, elevation, richness), player faction level, and save state. Individual explorable definitions are in `data/explorables/`.

## Technology Tree

Defined in `data/techs.lua` and split across `data/tech/tech_*.lua` by faction (robots, blight, alien, human, virus). Tech categories are in `data.tech_categories`. Each tech has `require_tech`, `unlocks` (array of ids it enables), and an `uplink_recipe`.

## Scenarios

Each scenario (in `scenarios/`) extends the base `Data` package:
- `freeplay.lua` — primary game mode; includes tutorial, events, codex entries, and explorable config for the open world
- `versus.lua` — PvP with fixed maps (map data in `scenarios/map_data/`)
- Challenge scenarios (`registers1`, `registers2`, `signals1`, `nomad1`, `towerdefense`, `resimulator`) — self-contained puzzles with specific win conditions

## UI Architecture (`ui/`)

`ui/ui.lua` is the entry point. Key subsystems:
- `widgets.lua` / `utilities.lua` — base UI helpers
- `Program.lua` — visual behavior/instruction editor
- `BlueprintEditor.lua` / `LinkEditor.lua` — blueprint and register linking UI
- `Library.lua` — saved behavior library management
- `FrameView.lua` / `BuildView.lua` — entity inspection and build menus
- `Tech.lua` / `Codex.lua` — tech tree and in-game encyclopedia

Input bindings use `Input.BindAction(name, event, handler)`. Default bindings are in `InputDefaultActionMappings` at the bottom of `ui/ui.lua`.

## `blz-desynced-toolkit` (sibling repo `../blz-desynced-toolkit/`)

The wire codec, BSF pipeline, and Lua-backed runtime/interpreter live in the separate `blz-desynced-toolkit` package, imported here as `blz.desynced_toolkit`. **Its own `CLAUDE.md` (in that repo) is authoritative** for module docs, the reuse-real-Lua standing convention, its test suite, and the CLI (`desynced-bsf`, also runnable as `python -m blz.desynced_toolkit.bsf`) — read that file, not a summary here, before touching anything in that layer.

This repo depends on it via a `tool.uv.sources` local path dependency (see `pyproject.toml`) pointing at the sibling directory — `uv sync` here picks up changes made there immediately (editable install), no publish step needed. If a change to the toolkit is needed to support work in this repo, make it in `../blz-desynced-toolkit/`, following that repo's own conventions and running its test suite there before returning here.

**Referring to nodes and ids when talking to the user:** BSF node ids (`n27`) are decompiler-generated — invisible in the game's own editor and unstable across re-exports (the editor reorders untouched nodes on every save) — and internal game ids are often just as invisible (`c_radar` displays as "Long-Range Radar", `c_adv_miner` as "Laser Mining Tool", `v_broken` as "Heavily Damaged"). Never anchor a user-facing statement on a bare node id, and pair any non-obvious internal id with its in-game name. Phrase references in the vocabulary the user actually sees: the instruction's **visual display name** (`set_reg` displays as "Copy" — `ArgCache.display_name(op)`), id display names (`ArgCache.id_display_name(id)`, or `desynced-bsf ids <query>` to search both directions), configured arguments, `cmt`, and the enclosing `label` section — e.g. "the Copy that writes `v_resource` into `$NextTag`, right after the Is Moving check in the 'Setup search' section", not "n24". `decompile --annotate` inlines all of these as `#` comments. Quote a short BSF excerpt when discussing wiring, so the user can correlate surrounding instructions with what's on their screen. Bare node ids and internal ids are fine (and correct) *within* BSF text itself and in tool output the user asked to see.

Known gaps and deferred work for all of the above are tracked in `todo.md`, not here.

## Design Docs Built on This Data

All at the workspace root. Each doc carries its own status/open-items section — trust those and `todo.md` over any summary here. The tooling-facing spec docs (`instructions_index.md`, `behavior_format.md`, `behavior_source_format.md`, `mock_world_spec.md`) moved to `../blz-desynced-toolkit/` with the code they document — see that repo's `CLAUDE.md`. The docs below are personal behavior-design work, not tooling docs, and stay here:

- `combat_squad_spec.md` — behavior design spec for squad AI (v2: Captain/Gunner/Healer/Power-Provider), rebuilt from real squad experiments around one engine fact: a weapon's manual target only takes effect while the faction can *see* it, so an unarmed high-visibility Captain personally maintains the vision lock from standoff, gates the squad through a rally point (the anti-scatter core), and issues one-value commands over its own signal register. The v1 Beacon-era design and its `beacon.dcs` implementations are git history (the `.dcs` files remain as test fixtures). All four roles are implemented in BSF (`library/`) and have run live for several hours across mixed frame/turret loadouts — see the spec's own open items and `todo.md` for the loss pattern still being tuned.
- `hex_expansion_math.md` — integer-only coordinate math for a hexagonal-spiral expansion behavior (`HexAt(origin, R, T) -> coord` and inverse `HexIndexOf`). Both are validated against closed-form reference math through the real `data/instructions.lua` (`../blz-desynced-toolkit/tests/test_hex_expansion.py`, `test_hexindexof_compiled.py`), and `HexAt` is validated in-game (that repo's `tests/data/hexat_test.dcs`, `HexIndexOf_test_1.dcs` bundle the sub-behaviors with self-test harnesses). The in-game `.dcs` and the later readability-driven redesign have not been reconciled — see the doc's "Open items".
- `blight_magnifier_mining.md` — two-part design doc: the real component-efficiency math for Blight Magnifier building layouts (overclock boost is building-wide and additive, the oversized-socket +50% is per-component, so 2+ magnifiers with a Large socket require per-slot brute-force enumeration), and the `MinerDrone`/`MagnifierSignal` behaviors authored directly in BSF. Its later sections record the in-game-tested findings, including a reversed earlier claim (`SetStateStartWork` applies Overclock/`eff_boost` natively even when a component's Lua never reads `get_work_time`) and the empirically-settled lattice density. The deployed behaviors live in `library/`.
- `observer_redesign.md` — the Observer behavior's redesign into two concurrently-advancing state machines (sensing priority-ladder + movement), and the `Async Radar` subroutine's interface history, ultimately split into `Async Radar Set`/`Async Radar Get` so arming and reading are independent calls. Documents real engine semantics found along the way (`event_parameter`/`event_radio` hijack the whole program counter; `c_portable_radar`'s fixed `TICKS_PER_SECOND` re-arm quirk). The deployed behaviors live in `library/`.

## `.dcs`/`.bsf` file locations

- `library/` — reference copies of real, actively-used-in-game behaviors/blueprints, re-exported from the in-game library after live editing sessions. These are living documents the user re-exports after further in-game edits, not one-time snapshots — check `git log` on a file for its history, and don't assume the checked-in version is final. **Stored as BSF text (`.bsf`), not raw `.dcs`** (since 2026-07-22): a `.dcs` re-export is a single base62 blob, so any edit — even a cosmetic re-save — produces a meaningless one-line git diff; BSF text diffs readably. **A directory-level by-reference store** (`blz.desynced_toolkit.bsf.library`, since 2026-07-22), not a flat pile of self-contained files: a sub-behavior shared by more than one caller (`Async Radar Set`/`Async Radar Get`, called from both `observer.bsf` and `mining_leader.bsf`) lives in its own file, referenced by name (`sub Async Radar Set from "async-radar-set.bsf"`) instead of duplicated — mirroring the game's own by-reference saved-library semantics, where editing a shared sub updates every caller. Update flow: copy the behavior from the in-game library to the clipboard, then `desynced-bsf import library --input -` (recursively splits any embedded sub into its own file, updating one already there if its content changed — reported via a real semantic diff, not silently — and flagging any *other* library file whose reference now points at content that just changed). Two independently-imported behaviors that happen to embed the *same-named* sub with genuinely different content is exactly the by-reference staleness this store is meant to expose, not a bug in the tool — resolve it by checking which content is actually current (e.g. re-copy that behavior fresh from the game) before re-importing, not by picking whichever import happened to run last. To push a library behavior back into the game (after a hand-edit, or one authored fresh in BSF), `desynced-bsf export library <name>` resolves every reference back into an embedded `dependencies` array and compiles straight to `.dcs`, ready to paste — note this does **not** preserve the editor's hand-arranged node layout (`nx`/`ny` positions aren't modeled in BSF yet), so a compiled-and-repasted behavior's nodes land in default positions even though the logic is unchanged (an accepted tradeoff, user decision 2026-07-22 — auto-layout is what's actually used day to day). Blueprints (wire type `'B'`, e.g. the magnifier lattice) are not BSF-decompilable and stay as `.dcs` — inspect them with `LupaEngine.decode_dcs` directly.
- `tests/` — this repo's own (small) test suite: personal `library/` behaviors exercised against the `blz-desynced-toolkit` dependency (e.g. lint-clean checks across the whole library, a mock-world run of the deployed HexAt). Fixture-heavy toolkit tests (the wire codec, BSF round-trips, the interpreter/mock-world engine itself) live in `../blz-desynced-toolkit/tests/` instead, with their own `tests/data/`.
- Workspace root — any `.dcs`/`.bsf` here is a temporary paste-to-view scratch file, **never committed** (user: committing behavior files at the root is bad practice — everything checked in belongs in `library/` or, for tooling fixtures, the toolkit repo's own `tests/data/`).

## Corpus & Analysis Tooling (`scripts/`, `corpus/`)

Built to ground `behavior_source_format.md`'s design decisions in real usage rather than hand-picked examples. Not covered by `uv run pytest tests/` — standalone scripts, exercised by direct manual runs; a coverage gap tracked in `todo.md`.

- `scripts/collect_clipboard_corpus.py` — watches this environment's clipboard (the dev machine is a VirtualBox guest of the actual Windows gaming host, with a shared clipboard) and harvests any `.dcs`-shaped string that appears, capturing paired context (surrounding thread text) alongside each decoded item. Meant to run by hand while browsing the official Desynced Discord forum channels.
- `scripts/collect_steam_forum.py <discussion_url>` — scrapes a Steam Community discussion thread (plain server-rendered HTML, paginates via `?ctp=N`, no login/JS needed) for shared `.dcs` strings, pairing each with its comment author/text.
- `corpus/` (gitignored — bulk redistribution of other players' shared creative work, kept local-only) — the collected decoded behaviors/blueprints, split by source (`discord_behaviors/`, `steam_behaviors/`, `github_fixtures/` — the last from `ribrdb/desynced-tools`'s test data only; that repo's conversion *logic* is explicitly not trusted or reused, see project memory). Each source dir has `index.jsonl` and a `context/` subdirectory.
- `scripts/analyze_corpus.py` — the statistical analysis behind `behavior_source_format.md`'s empirical justification: finds behaviors generically wherever they occur (top-level, embedded in a blueprint's `c_behavior`, nested under `subs`/`dependencies`) and reports control-flow fan-in, `jump`/`label`/`call`/loop usage, and data-variable fan-out across the corpus.
- `scripts/render_examples.py <dcs_file> <behavior_path> <out_prefix>` — thin CLI wrapper over `blz.desynced_toolkit.bsf`: decompiles a real behavior and writes `<out_prefix>.bsf.txt` plus one `<out_prefix>.<i>.mmd` Mermaid file per component.
