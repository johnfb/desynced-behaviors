# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

**End goal, stated explicitly because it governs prioritization:** enable Claude Code to do *agentic programming* of Desynced behaviors — read, review, and edit a real in-game behavior the way it would work with source code in a normal IDE, then write the result back into the live game. Everything else in this repo (the `.dcs` wire codec, the BSF pipeline, the Lua-backed interpreter) is **instrumental infrastructure in service of that goal**, not the goal itself.

Both halves of that goal have working answers:

- **Transport**: Desynced's own copy/paste (`Tool.GetClipboard`/`SetClipboard`, producing the `.dcs` strings this repo's codec decodes) plus a VirtualBox host/guest shared clipboard gives a live, bidirectional read/write channel with zero game-side code. The `python -m desynced_toolkit.bsf` CLI is built to sit in a shell pipeline against that clipboard.
- **Representation**: Desynced behaviors are genuine flow graphs (arbitrary-fan-in control edges, arbitrary-fan-out data edges — confirmed against real user behaviors, not just synthetic ones generated by this project's own tooling), so the adopted representation is BSF, a graph-native text format (`behavior_source_format.md`, implemented in `desynced_toolkit.bsf`) — not tree-structured pseudocode, which was tried first (a Python-subset `AstCompiler`, removed 2026-07-14; git history only) and abandoned as a lossy fit.

This repo is tooling, documentation, and design work built *on top of* an extract of the base game data assets for **Desynced** (by The Desynced Team), a factory/automation game with programmable robots — the Lua scripts, textures, sounds, and other assets that define game behavior. There is no build system for the game data itself — it's a read-only reference for analyzing and understanding game mechanics.

**The game data extract lives outside this repo.** You need your own local copy of it; this repo does not include or fetch it. **Convention:** place it as a sibling directory of this repo, named `desynced-game-data` (i.e. if this repo is at `/path/to/desynced/`, the extract goes at `/path/to/desynced-game-data/`) — everything in this repo that looks for it (tests, tooling) defaults to that location with no configuration. If you put it somewhere else, set the `DESYNCED_GAME_DATA` env var to its path. Every path mentioned in this doc as if it were local (`data/instructions.lua`, `ui/ui.lua`, `main.zip`, etc.) is **relative to that external directory**, not this repo's root. Both Bash and Read can reach it directly via absolute path without any special setup; use `/add-dir <path-to-your-extract>` if a given session needs it added explicitly. `desynced_toolkit.assets.open_asset_source(...)` (this repo's own code) takes that directory (or its `main.zip`) as a plain argument — nothing is hardcoded.

The entry point manifest is `def.json` (in the external directory), which defines all packages and their load order.

**Current/planned work lives in `todo.md`** (checked in, preferred over session-scoped task tools) — read it rather than reconstructing project state from this file or git history.

## Package Architecture

The game uses a package system. Each package has an `entry` Lua file that sets `package.includes` (a list of files to load in order) and lifecycle callbacks:

- `package:init()` — runs at startup, registers data definitions into global tables
- `package:init_ui()` — UI-only initialization
- `package:setup_scenario(settings)` — called before scenario starts
- `package:validate_data()` — dev-mode data integrity checks
- `package:on_update(ver)` — migration logic for saved games upgrading from old versions

The packages defined in `def.json`:

| Package | Entry | Type | Purpose |
|---|---|---|---|
| `Data` | `data/data.lua` | Data | All game definitions (items, frames, components, etc.) |
| `UI` | `ui/ui.lua` | UI | In-game HUD, menus, input bindings |
| `FrontEnd` | `frontend/frontend.lua` | Scenario | Main menu and lobby |
| `Freeplay` | `scenarios/freeplay.lua` | Scenario | Primary open-world game mode |
| `Versus` | `scenarios/versus.lua` | Scenario | PvP scenario |
| `Resim/Registers1/Registers2/Signals/Nomad/TowerD` | `scenarios/*.lua` | Challenge | Puzzle/challenge scenarios |

## Core Data Definitions (`data/`)

All game definitions are registered into global `data.*` tables. Each definition file documents its own schema at the top as a comment block.

**Key definition tables:**

- `data.frames` — Entity "frames" (the physical body of a unit or building). Built with `Frame:RegisterFrame(id, def)`. Frame definitions include movement speed (non-zero = unit/bot), health, slots, initial components, and lifecycle callbacks (`on_placed`, `on_destroy`, `on_interact`).
- `data.components` — Attachable components that go into frame sockets. Define behavior, production recipes, power consumption, registers, and event callbacks (`on_add`, `on_remove`, `on_update`, `on_trigger`, `on_take_damage`).
- `data.items` — Inventory items with stack sizes and recipes.
- `data.instructions` — The visual programming instruction set. Each instruction has `func(comp, state, cause, ...)` implementing its behavior. Instructions are categorized: Flow, Unit, Global, Math, Move, Component, AutoBase.
- `data.behaviors` — Pre-encoded behavior programs (stored as base62-encoded strings).
- `data.techs` — Technology tree nodes with uplink recipes and unlock lists.
- `data.explorables` — Procedurally placed world events/encounters. Each explorable implements `GetRelevancy(x, y, info)` and `SpawnExplorable(x, y)`.
- `data.values` — Enum-like named values (colors, radar filter types, alien signals).
- `data.visuals` / `data.visualassets` — Visual representations and socket positions.
- `data.fx` — Sound/particle effects.
- `data.biomes`, `data.landfeatures`, `data.cliffs` — World generation parameters.
- `data.puzzles` — Minigame puzzle definitions.

**Recipe helpers** (defined in `data/utilities.lua`):
- `CreateProductionRecipe(ingredients, producers, amount)` — manufactured items/components
- `CreateMiningRecipe(miners)` — mined resources
- `CreateUplinkRecipe(ingredients, ticks)` — research via uplink
- `CreateConstructionRecipe(ingredients, ticks)` — placed buildings

## Faction & Entity System

The game world is divided into **factions**: `player` (human-controlled), `world` (neutral explorables), `bugs`, `alien`, `anomaly`. Trust relationships between factions determine hostility. Faction-level data persists in `faction.extra_data`.

**Entity registers** are the core of the programming system. Frames have a fixed number of registers (FRAMEREG_GOTO, FRAMEREG_STORE, etc.) that hold typed values (entities, items, numbers, signals). The visual behavior editor programs these by constructing instruction sequences.

**Behaviors** are programs attached to components (particularly `c_behavior` and `c_autobase`). The `data/library.lua` handles compiling encoded behavior strings into bytecode (`GetFactionBehaviorAsm`) and caching them.

## Action System

The game uses typed action dispatchers:

- `PlayerAction.*` — actions sent by a player, validated server-side
- `EntityAction.*` — actions applied to a specific entity
- `FactionAction.*` — actions applied to a faction
- `MapMsg.*` — broadcast map-wide messages
- `UIMsg.*` — UI-only callbacks (run only on local client)
- `Chat.*` — multiplayer chat message handlers
- `Delay.*` — deferred map-tick callbacks
- `MapRun.*` — simulation-side broadcast

## Explorable System

Explorables are the world's discoverable encounters. The file `data/explorables.lua` contains `Explorable_SpawnAt(x, y, discoverer, filter)` which selects an explorable by weighted `GetRelevancy` scores based on tile data (blightness, elevation, richness), player faction level, and save state. Individual explorable definitions are in `data/explorables/`.

## Technology Tree

Defined in `data/techs.lua` and split across `data/tech/tech_*.lua` by faction (robots, blight, alien, human, virus). Tech categories are in `data.tech_categories`. Each tech has `require_tech`, `unlocks` (array of ids it enables), and an `uplink_recipe`.

## Scenarios

Each scenario (in `scenarios/`) extends the base `Data` package:
- `freeplay.lua` — primary game mode; includes tutorial, events, codex entries, and explorable config for the open world
- `versus.lua` — PvP with fixed maps (map data in `scenarios/map_data/`)
- Challenge scenarios (`registers1`, `registers2`, `signals1`, `nomad1`, `towerdefense`, `resimulator`) — self-contained puzzles with specific win conditions

## UI Architecture (`ui/`)

`ui/ui.lua` is the entry point. Key subsystems:
- `widgets.lua` / `utilities.lua` — base UI helpers
- `Program.lua` — visual behavior/instruction editor
- `BlueprintEditor.lua` / `LinkEditor.lua` — blueprint and register linking UI

- `Tech.lua` / `Codex.lua` — tech tree and in-game encyclopedia

Input bindings use `Input.BindAction(name, event, handler)`. Default bindings are in `InputDefaultActionMappings` at the bottom of `ui/ui.lua`.

## `desynced-toolkit` (`pyproject.toml`, `src/desynced_toolkit/`)

A Python package (managed with `uv`) whose runtime semantics are backed by `lupa` (an embedded real Lua interpreter) running the **actual, unmodified `data/instructions.lua`** rather than hand-reimplemented-in-Python semantics — hand reimplementation was the source of most bugs found earlier in this project. Everything past the wire-bytes layer works with **genuine Lua tables** (1-based, via `lupa`) rather than a Python dict standing in for one.

**Setup:** `uv sync` (installs `lupa` + dev deps into the gitignored `.venv/`). Needs the external game-data extract at the sibling-directory location above — tests skip cleanly if it's missing and `DESYNCED_GAME_DATA` isn't set.

**Lua version pin:** every module that touches a `LuaRuntime` imports `lupa.lua54 as lupa`, never bare `import lupa`. The shipped game embeds Lua 5.4 exactly (confirmed by disassembling `Desynced.exe`'s embedded Lua version string), while bare `lupa` resolves to whatever Lua it was built against (a newer version in current releases). A partial pin is worse than none: `lupa.lua_type()` from one version's submodule silently returns `None` (not an error) on a table produced by a *different* version's `LuaRuntime`, so a stray bare import breaks table-detection silently rather than loudly.

**Tests:** `uv run pytest tests/` (a few seconds). Run it after any change here before concluding something works — it's what actually backs the claims in this section. Tests read fixtures only from `tests/data/`; the suite covers the wire codec (decode/encode round-trips of every fixture), the BSF pipeline (IR and text round-trips, validation/lint, end-to-end decode→edit→recompile, Mermaid rendering, semantic diff), the real-Lua runtime/interpreter, and the hex-expansion math against closed-form references.

**Standing convention — reuse the real Lua, both data and logic:** don't reimplement in Python anything the actual game already does. Two parts: (1) *data shape* — build genuine Lua tables (1-based, via the running `lupa.LuaRuntime`) for anything the game itself represents as a Lua table, never a Python dict/list rendering; (2) *logic/decisions* — if a real function in the extract already makes some decision (a comparison, a branch, an addressing calculation), call it and use its actual result rather than re-deriving the decision in Python from values you happened to read back. Neither is about performance or purity: an extra representation/logic layer standing in for the real thing is where this project's bugs have come from, repeatedly.

### Modules

- `assets.py` — loads package files from either an extracted directory or directly out of a zip (e.g. `main.zip`) via stdlib `zipfile`, no extraction to disk needed *for the library*. (This is a Python-level abstraction only: Claude's own Read/Grep/Bash tools still can't see inside a zip, so ad hoc extraction remains the move for interactive exploration in conversation.)
- `dcs_wire.py` — the base62 + optional-zlib + custom-MessagePack-variant codec for Desynced Clipboard Strings ("DCS", this project's own coined term — the wire prefix is `DS` + a one-character type tag: `'C'` = behavior/code, `'B'` = blueprint, others uninvestigated). `decode_dcs`/`encode_dcs` build/consume genuine Lua tables. The real in-game conversion (`Tool.GetClipboard`/`SetClipboard`, `ui/Library.lua`) is engine-native — confirmed both by the absence of any Lua source for it and by disassembling `Desynced.exe` and structurally tracing the native decode worker against this module (see project memory for the trace; it also surfaced native `Tool.*` functions this project doesn't use yet: `EncodeEntityStates`, `EncodeTrustLevels`, `GetTableDelta`/`ApplyTableDelta`, `TableToString`/`StringToTable`, `CompareStringNoCase`, `ContainsStringNoCase`, `MatchLocalizedRichText`, `GetDateStr`, `GetTimeDurationStr`, `ParseEntityStates`). Also cross-validated against the official [StageGames/DesyncedJavaScriptUtils](https://github.com/StageGames/DesyncedJavaScriptUtils) `dsconvert.js` reference (two real bugs found in *their* encoder, documented in this module's docstring, deliberately not reproduced). It does **not** use a standard MessagePack library (repurposes type bytes `0xc1`/`0xc4`/`0xc5` for non-standard meanings) and has no instruction-metadata layer — string ⟷ Lua-table transport only.
- `engine_stub.lua` / `lua_runtime.py` — the `lupa`-backed runtime. Key fact, documented in both files: `Get`/`Set`/`GetNum` are **not** external engine globals — they're local aliases *within* `instructions.lua` itself for real `InstGet`/`InstSet`/`InstGetNum`/`InstBeginBlock` functions also defined in that file, resolving arguments through a `GetStack`-based unified integer addressing scheme (`state.mem[]` for locals/literals, `comp.owner:GetRegister()` for frame registers). These are reused **verbatim, unmodified** — only the genuinely-external pieces are stubbed: the `Value` type's arithmetic (confirmed engine-side), a `comp`/`comp.owner`/`Tool.NewRegisterObject` stand-in, `Tool.Hash`/`Map.GetSave` (the library-import path's dedup key and id counter), and small lifecycle surface (`Activate`/`Shutdown`, sleep). The real **`data/library.lua` loads too** — before `components.lua`/`instructions.lua`, both of which capture load-time local aliases of `GetCachedBehaviorAsm`/`GetFactionBehaviorAsm` — so behavior compilation/caching/import are the genuine game functions, not stubs. By default `LupaEngine` also loads the real Data-registry files (`utilities/library/values/items/components/frames`, the empirically-minimal subset) before `instructions.lua` and builds `data.all` — so `data.frames`/`data.components`/`data.items`/`data.values`/`data.all` are populated with genuine game defs and `FilterEntity`/`PrepareFilterEntity` run for real (the mock world's foundation, `mock_world_spec.md` Phase 0); pass `load_data_registries=False` for the bare instructions-only runtime (no `Interpreter` there — the behavior runtime needs the registries). `LupaEngine.decode_dcs`/`encode_dcs` wrap `dcs_wire.py`.
- `behavior_runtime.lua` / `interpreter.py` — runs a behavior given as a genuine Lua table (the shape `decode_dcs` hands back, or `bsf.compile_behavior`'s output) through the **real game behavior machinery**: the real `GetFactionBehaviorAsm` compiles it (memory interning, branch encoding, `make_asm` hidden args), the real `UploadBehavior`/`SetBehavior` install and start it on a component (dependency unpack with `call.sub` remapping, content-hash dedup into `faction.extra_data.library`; **parameters are component registers**, per the real `state.stk` addressing), and `BehaviorRuntime.Activate` — a port of the dispatch loop inside the real `c_behavior:on_update`, delegating every dead end to the real `c_behavior_on_end` — executes it, so the block stack (`InstBeginBlock`), `call`/return (genuine by-reference parameters; memory arrays shared across the call stack), `jump` label scans, and every branch decision are the game's own code. The Python `Interpreter` class is only *activation scheduling* (the part that is engine-native C++ in the real game): sleep countdown (`SetStateSleep(N)` = resume N ticks later, pinned by the dispatcher's own locked-mode source), movement-wake, and the run/run_ticks drivers. One deliberate harness deviation, documented in both files: a top-level fall-off ("restart") halts the run so tests terminate — the real engine restarts from Program Start forever, `exit` being the only genuine halt. (An earlier Python-simulated tier — own block stack, per-instruction arg translation, no `call` — is git history only.)
- `world.lua` / `mock_world.py` — the mock world (`mock_world_spec.md`, Phases 0–3 done): a populated, steppable environment where the real instruction funcs run against mock entities whose `.def` IS the real `data.frames`/`data.components` def. `world.lua` supplies only the engine-native leaves (spatial registry + floored-Euclidean `Map.GetDistance`/range gates, tiles, vision/trust, register banks, movement resolution — its movement-section header documents each rule's measured provenance; arrival tolerance is provisional pending the `arrival_probe` in-game run); `MockWorld` (Python) orchestrates: `spawn`/`add_component`/`set_tile`/`attach_behavior`/`step(n)` (behaviors → movement → deferred, per tick) plus a tick-stamped, entity-attributed `debug_print` capture stream (`prints`). Validated end-to-end by `test_movement_circuit_golden.py`: the real in-game movement-circuit behavior reproduces its own in-game log in the mock (exact tile sequence; tick deltas within the documented tolerance).
- `bsf/` — the working, bidirectional implementation of `behavior_source_format.md`'s BSF grammar, and the project's primary read/edit surface. `decompile.py` (real Lua table → `BsfBehavior` graph IR; rejects non-behavior wire types — a blueprint's `'B'` shape is not decompilable), `compile.py` (IR → real Lua table, resolving node ids to wire positions fresh on every call), `render_text.py`/`parse_text.py` (IR ⟷ BSF text), `render_mermaid.py` (IR → Mermaid flowcharts), `semantic_diff.py` (wire-position-independent diff between two saves — the real editor re-encodes untouched nodes' branch encoding and relative order on every save, so a raw text diff overreports), `values.py`/`ir.py` (the value/node/behavior dataclasses), `lint.py` (legal-but-suspicious checks — see below), `argcache.py` (live-`data.instructions[op].args`-backed arg-position/exec-branch resolution; `next_pin_name(op)` reads the real `exec_arg` field and is the single source of truth for the top-level pin's name/existence on both render and parse sides; also the game-id registry scan backing id validation and the id ⟷ display-name mapping). Validated by round-tripping every real fixture in `tests/data/` at both the table and text layers, plus an end-to-end decode → edit-as-text → recompile → re-encode exercise. Branch dead-ends render as `POP`: not a halt but a pop of the current context frame (innermost active loop or `call`), with an empty stack auto-restarting from Program Start — `exit` is the only genuine halt. The text layer is **strictly validating** (unknown ops/args/pins/ids/targets and duplicates all rejected with line numbers and suggestions — bare identifiers are checked against the real game registries, so a forgotten `$` sigil fails loudly), an op with 2+ exec pins must have **every** pin written (`>node`, `>POP`, or the explicit-fallthrough token `>NEXT`), and `lint.py` (run on every CLI compile) flags the legal-but-suspicious tier (unreachable nodes, literal jumps with no matching label, undeclared parameter slots) — see `behavior_source_format.md` § "Explicit-pin rule" and § "Validation" for the rules and the silent-miswire history behind them. **Explicitly not attempted, not just deferred:** static resolution of what a `POP`'s destination actually is inside a loop's dynamic scope — genuinely undecidable in general (halting-problem-equivalent). `render_mermaid` returns one diagram per **component**, found by forward reachability from Program Start (not undirected connectivity — a static `jump` back to a hub must not weld an unreachable-forward section into the main component); every declared exec pin renders as its own labeled edge, an edge leaving a component renders as a small local external-reference marker, and `direction`/`connect_resolved_jumps` are real parameters — see `behavior_source_format.md`'s "Visualization" section for the reasoning.
- `bsf/__main__.py` — makes the package runnable as `python -m desynced_toolkit.bsf {decompile,compile,semantic-diff,lint,ids}`, reading stdin/writing stdout (`--input`/`--output` also accepted), resolving game data the same way as the tests. Built to sit in a shell pipeline against the game's own clipboard, e.g. with `cb` a user-side wrapper for `xclip -selection clipboard`: `cb -o | python -m desynced_toolkit.bsf decompile > mybehavior.bsf`, hand-edit, then `python -m desynced_toolkit.bsf compile < mybehavior.bsf | cb -i` (`compile` also prints lint warnings to stderr). `decompile --annotate` adds `#` comments with in-game display names for correlation with the editor; `lint` runs the suspicious-construct checks standalone on BSF text or a `.dcs` string; `ids <query>` searches the internal-id ⟷ display-name mapping in both directions. Only a behavior/program clipboard item (wire type `'C'`, the editor's "Copy Program" action) is decompilable; a blueprint (`'B'`) is rejected with an error — use `LupaEngine.decode_dcs` directly to inspect one.

**Referring to nodes and ids when talking to the user:** BSF node ids (`n27`) are decompiler-generated — invisible in the game's own editor and unstable across re-exports (the editor reorders untouched nodes on every save) — and internal game ids are often just as invisible (`c_radar` displays as "Long-Range Radar", `c_adv_miner` as "Laser Mining Tool", `v_broken` as "Heavily Damaged"). Never anchor a user-facing statement on a bare node id, and pair any non-obvious internal id with its in-game name. Phrase references in the vocabulary the user actually sees: the instruction's **visual display name** (`set_reg` displays as "Copy" — `ArgCache.display_name(op)`), id display names (`ArgCache.id_display_name(id)`, or `python -m desynced_toolkit.bsf ids <query>` to search both directions), configured arguments, `cmt`, and the enclosing `label` section — e.g. "the Copy that writes `v_resource` into `$NextTag`, right after the Is Moving check in the 'Setup search' section", not "n24". `decompile --annotate` inlines all of these as `#` comments. Quote a short BSF excerpt when discussing wiring, so the user can correlate surrounding instructions with what's on their screen. Bare node ids and internal ids are fine (and correct) *within* BSF text itself and in tool output the user asked to see.

Known gaps and deferred work for all of the above are tracked in `todo.md`, not here.

## Design Docs Built on This Data

All at the workspace root. Each doc carries its own status/open-items section — trust those and `todo.md` over any summary here.

- `instructions_index.md` — auto-generated reference of every entry in `data.instructions` (`data/instructions.lua`): visual-editor name, op id, and argument list (in/out/exec, in declaration order, with filter type tags). Regenerate by re-parsing that file if it changes.
- `behavior_format.md` — the reverse-engineered spec for the *wire* format a behavior decodes into/from: register/slot addressing, branch and fall-through resolution, `make_asm`/hidden-literal fields, variable-length-arg instructions, stopping semantics. Together with `instructions_index.md`, it's the layer between the wire bytes (transport only, no instruction semantics) and authoring a behavior; read both before writing or editing any `.dcs` content by hand, or before answering *any* question about register/value semantics, branch/jump encoding, or behavior control flow — not just when actively editing (see the gotchas index below). One historical caveat: where its prose mentions the retired `dsc_codec.py` or 0-based dict keys, that was the old tool's rendering choice — `dcs_wire.py` hands back the same information as genuine 1-based Lua tables.
- `behavior_source_format.md` — the BSF spec: a graph-native textual read/edit representation (nodes with explicit branch edges, not tree-structured pseudocode), adopted after corpus analysis of hundreds of real user behaviors showed tree syntax can't express real usage (`jump`/`label` computed dispatch alone is indirect goto by construction). Read it before implementing or reasoning about any decompiler/compiler work; its "Status" section records the real grammar gaps found and closed during implementation (parameter direction, hidden literal fields, `keepvars`/`keeparrays`, bare-bool args, `desc`, and more).
- `combat_squad_spec.md` — behavior design spec for squad AI (v2: Captain/Gunner/Healer/Power-Provider), rebuilt from real squad experiments around one engine fact: a weapon's manual target only takes effect while the faction can *see* it, so an unarmed high-visibility Captain personally maintains the vision lock from standoff, gates the squad through a rally point (the anti-scatter core), and issues one-value commands over its own signal register. The v1 Beacon-era design and its `beacon.dcs` implementations are git history (the `.dcs` files remain as test fixtures). Not yet implemented in BSF or tested in-game — see the spec's own open items and `todo.md`.
- `hex_expansion_math.md` — integer-only coordinate math for a hexagonal-spiral expansion behavior (`HexAt(origin, R, T) -> coord` and inverse `HexIndexOf`). Both are validated against closed-form reference math through the real `data/instructions.lua` (`tests/test_hex_expansion.py`, `tests/test_hexindexof_compiled.py`), and `HexAt` is validated in-game (`tests/data/hexat_test.dcs`, `HexIndexOf_test_1.dcs` bundle the sub-behaviors with self-test harnesses). The in-game `.dcs` and the later readability-driven redesign have not been reconciled — see the doc's "Open items".
- `blight_magnifier_mining.md` — two-part design doc: the real component-efficiency math for Blight Magnifier building layouts (overclock boost is building-wide and additive, the oversized-socket +50% is per-component, so 2+ magnifiers with a Large socket require per-slot brute-force enumeration), and the `MinerDrone`/`MagnifierSignal` behaviors authored directly in BSF. Its later sections record the in-game-tested findings, including a reversed earlier claim (`SetStateStartWork` applies Overclock/`eff_boost` natively even when a component's Lua never reads `get_work_time`) and the empirically-settled lattice density. The deployed behaviors live in `library/`.
- `observer_redesign.md` — the Observer behavior's redesign into two concurrently-advancing state machines (sensing priority-ladder + movement), and the `Async Radar` subroutine's interface history, ultimately split into `Async Radar Set`/`Async Radar Get` so arming and reading are independent calls. Documents real engine semantics found along the way (`event_parameter`/`event_radio` hijack the whole program counter; `c_portable_radar`'s fixed `TICKS_PER_SECOND` re-arm quirk). The deployed behaviors live in `library/`.
- `mock_world_spec.md` — design for extending `desynced_toolkit`'s `Interpreter` with a steppable **mock world** (mock entities a behavior can sense/command, stepped tick by tick) so multi-unit behaviors like the combat squad can be tested automatically. Same doctrine as the rest of the toolkit: run the real instruction funcs and reuse real Lua (`FilterEntity`/`PrepareFilterEntity`, the arg getters), mocking only the genuinely engine-native leaves (`Map.FindClosestEntity`, `comp:RequestStateMove`, `faction:IsSeen`, entity fields/methods); world state lives in Lua tables (entities carry the real frame/component `def`), Python-orchestrated. First version is sensing + movement (not combat); phased, design only — status in `todo.md`.

## `.dcs`/`.bsf` file locations

- `tests/data/` — fixtures the automated tests depend on, **checked in** (a fresh checkout must be able to run the suite). Some are snapshots of behaviors whose living copies are in `library/`; the two directories deliberately do not stay in sync. **This is also where purpose-built measurement/probe behaviors belong** (e.g. `movement_circuit_test.dcs` + its in-game log, `range_probe.bsf`/`.dcs`): such an instrument is a test with pending results, so it lives with the tests, wired into the suite, and its in-game run's numbers get checked in beside it as the golden data — never parked at the repo root.
- `library/` — reference copies of real, actively-used-in-game behaviors/blueprints, re-exported from the in-game library after live editing sessions. These are living documents the user re-exports after further in-game edits, not one-time snapshots — check `git log` on a file for its history, and don't assume the checked-in version is final. Blueprints (wire type `'B'`, e.g. the magnifier lattice) are not BSF-decompilable — inspect them with `LupaEngine.decode_dcs` directly.
- Workspace root — any `.dcs`/`.bsf` here is a temporary paste-to-view scratch file, **never committed** (user: committing behavior files at the root is bad practice — everything checked in belongs in `tests/data/` or `library/` per the rules above).

## Corpus & Analysis Tooling (`scripts/`, `corpus/`)

Built to ground `behavior_source_format.md`'s design decisions in real usage rather than hand-picked examples. Not covered by `uv run pytest tests/` — standalone scripts, exercised by direct manual runs; a coverage gap tracked in `todo.md`.

- `scripts/collect_clipboard_corpus.py` — watches this environment's clipboard (the dev machine is a VirtualBox guest of the actual Windows gaming host, with a shared clipboard) and harvests any `.dcs`-shaped string that appears, capturing paired context (surrounding thread text) alongside each decoded item. Meant to run by hand while browsing the official Desynced Discord forum channels.
- `scripts/collect_steam_forum.py <discussion_url>` — scrapes a Steam Community discussion thread (plain server-rendered HTML, paginates via `?ctp=N`, no login/JS needed) for shared `.dcs` strings, pairing each with its comment author/text.
- `corpus/` (gitignored — bulk redistribution of other players' shared creative work, kept local-only) — the collected decoded behaviors/blueprints, split by source (`discord_behaviors/`, `steam_behaviors/`, `github_fixtures/` — the last from `ribrdb/desynced-tools`'s test data only; that repo's conversion *logic* is explicitly not trusted or reused, see project memory). Each source dir has `index.jsonl` and a `context/` subdirectory.
- `scripts/analyze_corpus.py` — the statistical analysis behind `behavior_source_format.md`'s empirical justification: finds behaviors generically wherever they occur (top-level, embedded in a blueprint's `c_behavior`, nested under `subs`/`dependencies`) and reports control-flow fan-in, `jump`/`label`/`call`/loop usage, and data-variable fan-out across the corpus.
- `scripts/render_examples.py <dcs_file> <behavior_path> <out_prefix>` — thin CLI wrapper over `desynced_toolkit.bsf`: decompiles a real behavior and writes `<out_prefix>.bsf.txt` plus one `<out_prefix>.<i>.mmd` Mermaid file per component.

## Non-obvious engine/format gotchas — check before answering

These are confirmed-the-hard-way facts — each one was either misread from Lua source alone and corrected against real in-game behavior, or is a sharp edge easy to assume the "obvious" wrong thing about. `behavior_format.md` has the full detail, worked examples, and confirmation method for each (section names cited below; find them by heading, the file evolves). This index exists so the fact of a gotcha's existence surfaces even when the current question doesn't look like ".dcs editing" (e.g. a plain conceptual question about how register values work). Check the cited section before asserting any of these from memory, from the Lua source alone, or from general assumption:

- **Register/value composite semantics** — `coord`/`entity`/`id` are mutually exclusive on a value (setting one clears the other two); `num` is separate and coexists with any of them. `add`/`sub` (and `mul`/`div`, confirmed in-game to follow the same rules) combine them non-uniformly by operand shape: coordinate⊕coordinate adds both parts including `num`; coordinate⊕bare-number broadcasts the bare number onto both coordinate components and leaves the coordinate side's `num` untouched; entity/item⊕bare-number passes the entity through and adds `num` normally. § "Composite values and the `num` field".
- **Coordinate literal shape** — must be `{"coord": {"x": x, "y": y}}` (keyed object), never a `[x, y]` array — the array form round-trips fine through the codec but loads in-game as silently corrupted. § "Argument value encoding".
- **`exec`/`next` jump targets are raw 1-based Lua positions** — never adjusted to any 0-based rendering. § "Off-by-one".
- **`exit` genuinely halts the behavior** — nothing runs again until something else explicitly restarts it, despite `state.counter = 1; return true` reading from source alone like "yield and resume next tick." `restart` unconditionally jumps to Program Start (unwinding the *entire* subroutine call stack in one step if called from inside a `call`ed sub-behavior, not just returning one level) but does **not** yield — it can still spin forever under `unlock` with no other yield point. § "Stopping a behavior".
- **`next: false` inside a block (loop/`sequence`) pops to the *enclosing* block's own next-step logic**, not Program Start — only true top-level fall-off does that. § "Block-type instructions".
- **`check_number`'s "Equal" outcome isn't a numbered arg slot** — it's carried by the instruction's ordinary top-level `next`/fallthrough. To test equality, wire both `If Larger` and `If Smaller` to the same "not equal" target and let fallthrough be "equal". § "`check_number`'s \"equal\" case".
- **An omitted `exec` arg never consults the top-level `next`** — `If Larger`, `If Smaller`, and `next` (Equal) are three independent slots, each with its own fallthrough-to-next-instruction default; setting `next` does not also cover an omitted `If Smaller`/`If Larger`. § "Gotcha: an omitted exec arg never consults the top-level `next`".
- **`for_number`'s omitted `Step` auto-detects direction from `From`/`To`**, including flipping sign at runtime for a bound expression that can invert (e.g. `6*R - 1` at `R=0`) — this silently reverses direction and runs at least once, rather than running zero times the way a fixed-step loop would. § "`for_number`'s `Step` auto-direction".
- **`call`'s `sub` field** is a string (saved-library id), *or* a small positive int (1-based index into the top-level `dependencies` array for an embedded sub-behavior), *or* `-1` (recursive self-call sentinel, distinct from the legacy runtime's `0`) — never assume it's always a library id. § "`call`'s `sub`".
- **`keepvars`/`keeparrays` only ever apply to the outermost behavior assigned to the component, never to a `call`ed sub-behavior** — a subroutine's own local variables/memory arrays are always freshly allocated on every single `call` and unconditionally discarded on return, regardless of what its own `keepvars`/`keeparrays` say; even `restart`/`exit` fired from deep inside a nested call unwind the *entire* call stack first and consult only the outer program's own flags. Separately, **memory arrays are one shared table across the entire call stack within a run** (keyed by Index value, never windowed per call) — a subroutine and its caller using the same Index touch the *same* array. § "Top-level envelope".
- **An omitted `next`/exec-arg field is a real, explicit wire, not "no decision was made"** — the in-game editor's own compiler only omits it when there genuinely is a connection to the instruction placed immediately after (redundant with position, so it skips writing an int); a pin the user actually left unconnected always gets explicit `false` written, unconditionally. So "omitted" and "false" are never interchangeable-by-accident: two different authorial states, and only `false` means a genuine dead end. § "Branch and fall-through resolution"; `desynced_toolkit.bsf.argcache.resolve_branch`'s docstring has the full 3-way case analysis. (BSF's own text format treats fallthrough as positional and recomputes it fresh on every recompile — a considered choice for text editability, see `behavior_source_format.md` § "Node identity vs. wire position" — but only a single-pin op may leave it unwritten; an op with 2+ exec pins must spell every pin as `>node`/`>POP`/`>NEXT`, per § "Explicit-pin rule".)
- **...except that "omission is a deliberate wire" only holds for a pin that existed in the instruction's schema when the behavior was compiled** — instructions gain new exec pins across game updates (`domove`/`dodrop`'s "Path Blocked" pin post-dates some real saved behaviors), and old data never had that slot to omit deliberately. The current editor cannot distinguish "deliberately left unconnected" from "didn't exist yet" for old data, and doesn't try. The compiled *effect* is identical either way, but don't narrate an old behavior's omitted pin as a conscious authorial choice — it might just be schema age. § "Branch and fall-through resolution" (the "Real exception" callout).
- **A duplicate declared arg/pin name (e.g. `for_entities_in_range`'s three args all literally named "Filter") is disambiguated by occurrence count, not raw wire position** — `desynced_toolkit.bsf.argcache.arg_pin_names` renders them `Filter`, `Filter2`, `Filter3` in declaration order regardless of wire position. When reading BSF text, a numbered pin suffix means "which occurrence of this name," never "which wire position" (a position-based misreading once produced a false "the filter chain is broken" conclusion about a correct user behavior; regression test `test_duplicate_pin_names_disambiguated_by_occurrence_not_position`).
- **A jump/label id literal's `num` suffix is a real, load-bearing part of the target, not incidental** — real user behaviors reuse one label icon with different `num` values as genuinely distinct jump destinations (the editor has a limited fixed set of label icons). Always treat `(id, num)` together as the target identity, never `id` alone (regression test `test_jump_label_annotation_distinguishes_by_num`).
- **`REG_INFINITE`/`REG_NOT` are `-2147483648`/`-2147483647` (INT32-scale), not Lua's own 64-bit `math.maxinteger`/`math.mininteger`** — confirmed via a real in-game register-display test cross-checked against `ui/RegisterSelection.lua` three independent ways. Notably `REG_NOT` is the *smaller-magnitude* sentinel — an "opposite extremes" guess gets the ordering backwards. **An input pin left unconfigured in the editor displays blank there, but resolves at runtime to a real `num=0`** through `Get()`'s empty-register fallback — indistinguishable from an explicit `0` once copied into a register. § "Reserved sentinel numbers"; `engine_stub.lua`'s values match.
