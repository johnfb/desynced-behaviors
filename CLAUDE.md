# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

**End goal, stated explicitly because it governs prioritization:** enable Claude Code to do *agentic programming* of Desynced behaviors — read, review, and edit a real in-game behavior the way it would work with source code in a normal IDE, then write the result back into the live game. Everything else in this repo (the `.dcs` wire codec, the compiler, the Lua-backed interpreter) is **instrumental infrastructure in service of that goal**, not the goal itself. As of 2026-07-06 the actual game-to-Claude transport for this turned out to already exist for free — Desynced's own copy/paste (`Tool.GetClipboard`/`SetClipboard`, producing the same `.dcs` string this repo's codec decodes) plus a VirtualBox host/guest shared clipboard gives a live, bidirectional read/write channel with zero new game-side code. The remaining open problem is representation: Desynced behaviors are genuine flow graphs (arbitrary-fan-in control edges, arbitrary-fan-out data edges — confirmed against real user behaviors, not just our own generated ones), and forcing that into tree-structured pseudocode (this repo's current Python-like compiler target) is a lossy fit under active reconsideration — see `desynced-toolkit` below and project memory for where that stands.

This repo is tooling, documentation, and design work built *on top of* an extract of the base game data assets for **Desynced** (by The Desynced Team), a factory/automation game with programmable robots — the Lua scripts, textures, sounds, and other assets that define game behavior. There is no build system for the game data itself — it's a read-only reference for analyzing and understanding game mechanics.

**The game data extract lives outside this repo** (moved out on 2026-07-04 to stop ~560MB of vendored, untracked game assets from cluttering a tooling project — see `desynced-toolkit` below for the actual thing being built here). You need your own local copy of it; this repo does not include or fetch it. **Convention:** place it as a sibling directory of this repo, named `desynced-game-data` (i.e. if this repo is at `/path/to/desynced/`, the extract goes at `/path/to/desynced-game-data/`) — everything in this repo that looks for it (tests, tooling) defaults to that location with no configuration. If you put it somewhere else, set the `DESYNCED_GAME_DATA` env var to its path. Every path mentioned in this doc as if it were local (`data/instructions.lua`, `ui/ui.lua`, `main.zip`, etc.) is **relative to that external directory**, not this repo's root. Both Bash and Read can reach it directly via absolute path without any special setup; use `/add-dir <path-to-your-extract>` if a given session needs it added explicitly. `desynced_toolkit.assets.open_asset_source(...)` (this repo's own code) takes that directory (or its `main.zip`) as a plain argument — nothing is hardcoded.

The entry point manifest is `def.json` (in the external directory), which defines all packages and their load order.

## Package Architecture

The game uses a package system. Each package has an `entry` Lua file that sets `package.includes` (a list of files to load in order) and lifecycle callbacks:

- `package:init()` — runs at startup, registers data definitions into global tables
- `package:init_ui()` — UI-only initialization
- `package:setup_scenario(settings)` — called before scenario starts
- `package:validate_data()` — dev-mode data integrity checks
- `package:on_update(ver)` — migration logic for saved games upgrading from old versions

The packages defined in `def.json`:

| Package | Entry | Type | Purpose |
|---|---|---|---|
| `Data` | `data/data.lua` | Data | All game definitions (items, frames, components, etc.) |
| `UI` | `ui/ui.lua` | UI | In-game HUD, menus, input bindings |
| `FrontEnd` | `frontend/frontend.lua` | Scenario | Main menu and lobby |
| `Freeplay` | `scenarios/freeplay.lua` | Scenario | Primary open-world game mode |
| `Versus` | `scenarios/versus.lua` | Scenario | PvP scenario |
| `Resim/Registers1/Registers2/Signals/Nomad/TowerD` | `scenarios/*.lua` | Challenge | Puzzle/challenge scenarios |

## Core Data Definitions (`data/`)

All game definitions are registered into global `data.*` tables. Each definition file documents its own schema at the top as a comment block.

**Key definition tables:**

- `data.frames` — Entity "frames" (the physical body of a unit or building). Built with `Frame:RegisterFrame(id, def)`. Frame definitions include movement speed (non-zero = unit/bot), health, slots, initial components, and lifecycle callbacks (`on_placed`, `on_destroy`, `on_interact`).
- `data.components` — Attachable components that go into frame sockets. Define behavior, production recipes, power consumption, registers, and event callbacks (`on_add`, `on_remove`, `on_update`, `on_trigger`, `on_take_damage`).
- `data.items` — Inventory items with stack sizes and recipes.
- `data.instructions` — The visual programming instruction set. Each instruction has `func(comp, state, cause, ...)` implementing its behavior. Instructions are categorized: Flow, Unit, Global, Math, Move, Component, AutoBase.
- `data.behaviors` — Pre-encoded behavior programs (stored as base62-encoded strings).
- `data.techs` — Technology tree nodes with uplink recipes and unlock lists.
- `data.explorables` — Procedurally placed world events/encounters. Each explorable implements `GetRelevancy(x, y, info)` and `SpawnExplorable(x, y)`.
- `data.values` — Enum-like named values (colors, radar filter types, alien signals).
- `data.visuals` / `data.visualassets` — Visual representations and socket positions.
- `data.fx` — Sound/particle effects.
- `data.biomes`, `data.landfeatures`, `data.cliffs` — World generation parameters.
- `data.puzzles` — Minigame puzzle definitions.

**Recipe helpers** (defined in `data/utilities.lua`):
- `CreateProductionRecipe(ingredients, producers, amount)` — manufactured items/components
- `CreateMiningRecipe(miners)` — mined resources
- `CreateUplinkRecipe(ingredients, ticks)` — research via uplink
- `CreateConstructionRecipe(ingredients, ticks)` — placed buildings

## Faction & Entity System

The game world is divided into **factions**: `player` (human-controlled), `world` (neutral explorables), `bugs`, `alien`, `anomaly`. Trust relationships between factions determine hostility. Faction-level data persists in `faction.extra_data`.

**Entity registers** are the core of the programming system. Frames have a fixed number of registers (FRAMEREG_GOTO, FRAMEREG_STORE, etc.) that hold typed values (entities, items, numbers, signals). The visual behavior editor programs these by constructing instruction sequences.

**Behaviors** are programs attached to components (particularly `c_behavior` and `c_autobase`). The `data/library.lua` handles compiling encoded behavior strings into bytecode (`GetFactionBehaviorAsm`) and caching them.

## Action System

The game uses typed action dispatchers:

- `PlayerAction.*` — actions sent by a player, validated server-side
- `EntityAction.*` — actions applied to a specific entity
- `FactionAction.*` — actions applied to a faction
- `MapMsg.*` — broadcast map-wide messages
- `UIMsg.*` — UI-only callbacks (run only on local client)
- `Chat.*` — multiplayer chat message handlers
- `Delay.*` — deferred map-tick callbacks
- `MapRun.*` — simulation-side broadcast

## Explorable System

Explorables are the world's discoverable encounters. The file `data/explorables.lua` contains `Explorable_SpawnAt(x, y, discoverer, filter)` which selects an explorable by weighted `GetRelevancy` scores based on tile data (blightness, elevation, richness), player faction level, and save state. Individual explorable definitions are in `data/explorables/`.

## Technology Tree

Defined in `data/techs.lua` and split across `data/tech/tech_*.lua` by faction (robots, blight, alien, human, virus). Tech categories are in `data.tech_categories`. Each tech has `require_tech`, `unlocks` (array of ids it enables), and an `uplink_recipe`.

## Scenarios

Each scenario (in `scenarios/`) extends the base `Data` package:
- `freeplay.lua` — primary game mode; includes tutorial, events, codex entries, and explorable config for the open world
- `versus.lua` — PvP with fixed maps (map data in `scenarios/map_data/`)
- Challenge scenarios (`registers1`, `registers2`, `signals1`, `nomad1`, `towerdefense`, `resimulator`) — self-contained puzzles with specific win conditions

## UI Architecture (`ui/`)

`ui/ui.lua` is the entry point. Key subsystems:
- `widgets.lua` / `utilities.lua` — base UI helpers

Input bindings use `Input.BindAction(name, event, handler)`. Default bindings are in `InputDefaultActionMappings` at the bottom of `ui/ui.lua`.

## `desynced-toolkit` (`pyproject.toml`, `src/desynced_toolkit/`)

A real Python package (managed with `uv`), begun as a Python-syntax-like compiler/decompiler for behaviors, with runtime semantics backed by `lupa` (an embedded real Lua interpreter) running the **actual, unmodified `data/instructions.lua`** rather than hand-reimplemented-in-Python semantics — the source of most bugs found earlier in this project (see `behavior_format.md`'s revision history). Everything past the wire-bytes layer works with **genuine Lua tables** (1-based, via `lupa`) rather than a Python dict standing in for one — see `dcs_wire.py` below for why that's not just a style preference.

**Setup:** `uv sync` (installs `lupa` + dev deps like `pytest` from `pyproject.toml`/`uv.lock` into `.venv/`, which is gitignored and not checked in). Needs the external game-data extract, conventionally a sibling directory named `desynced-game-data` (see "What This Is" above) — tests skip cleanly if it's missing there and `DESYNCED_GAME_DATA` isn't set to point elsewhere.

**Lua version pin:** every module that touches a `LuaRuntime` imports `lupa.lua54 as lupa`, not bare `import lupa`. `lupa`'s own top-level module resolves to whatever Lua version it was built against by default — in the currently-installed `lupa` release that's **5.5**, not 5.4 — while the real shipped game (confirmed 2026-07-05 by disassembling `Desynced.exe`'s embedded Lua version string) embeds **Lua 5.4.4** exactly. `lupa.lua54` is a separate submodule bundling actual Lua 5.4 with the identical API surface (`LuaRuntime`, `lua_type`, ...), so aliasing the import is the entire fix — no other code changes needed. This matters because `lupa.lua_type()` from one version's submodule silently returns `None` (not an error) on a table produced by a *different* version's `LuaRuntime`, so a partial fix (pinning `LuaRuntime` in one file but leaving a bare `import lupa` for `lua_type` checks elsewhere) would silently break table-detection rather than fail loudly. All 684 tests pass unchanged after the pin, confirming no test in this suite happened to depend on a 5.4-vs-5.5 behavioral difference.

**Tests:** `uv run pytest tests/` (684 tests as of this writing, ~3.5s). Covers: real-Lua arithmetic/`check_number`/coordinate ops including `REG_INFINITE` (`test_lua_runtime.py`), `dcs_wire.py` decode/round-trip on every real `.dcs` file in the repo (`test_dcs_wire.py`), `HexAt`/`HexIndexOf` against closed-form reference math *and* round-tripped through each other, both executed via the real Lua interpreter (`test_hex_expansion.py`), and the compiler including the equality-handling regression (`test_compiler.py`). Run this after any change here before concluding something works — it's what actually backs the claims in this section, not just conversation history.

**Standing convention — reuse the real Lua, both data and logic:** don't reimplement in Python anything the actual game already does. This has two parts, both with real examples in this codebase: (1) *data shape* — build genuine Lua tables (1-based, via the running `lupa.LuaRuntime`) for anything the game itself represents as a Lua table, rather than a Python dict/list rendering (this is why `dcs_wire.py` replaced `dsc_codec.py`, see below); (2) *logic/decisions* — if a real function already exists in the extract that makes some decision (a comparison, a branch, an addressing calculation), call it and use its actual result, don't re-derive the same decision in Python from values you happened to read back (this is why `check_number`'s branch decision is now resolved by calling the real func with real target values, not by comparing numbers in Python — see `interpreter.py` below). Neither is about performance or purity: an extra representation/logic layer standing in for the real thing is where this project's bugs have come from, repeatedly.

- `assets.py` — loads package files from either an extracted directory or directly out of a zip (e.g. `main.zip`) via stdlib `zipfile`, no extraction to disk needed *for the library*. (This is a Python-level abstraction only: Claude's own Read/Grep/Bash tools still can't see inside a zip, so ad hoc extraction remains the move for interactive exploration in conversation.)
- `dcs_wire.py` — the base62 + optional-zlib + custom-MessagePack-variant codec for Desynced Clipboard Strings ("DCS", this project's own coined term — not an in-game or community name; renamed 2026-07-07 from this project's earlier "dsc", which was a misnomer: the wire prefix is just `DS` + a one-character type tag (`'C'` = behavior/code, `'B'` = blueprint, others uninvestigated), never literally "DSC" — see `dcs_wire.py`'s module docstring) — covering `data.behaviors` entries, player clipboard strings like `observer.dcs`. **Replaces the project's original standalone `dsc_codec.py` script** (deleted from the repo root — its logic lives here now), with one real change: `decode_dcs`/`encode_dcs` build/consume genuine Lua tables (via a `lupa.LuaRuntime`) instead of a Python `dict`/`list` rendering. This isn't just tidiness — `Tool.GetClipboard()`/`Tool.SetClipboard(item, type)` (`ui/Library.lua`) are the *real* functions that do this conversion in the actual game, confirmed engine-native (no Lua source for base62/msgpack/clipboard access anywhere in this extract), so a genuine Lua table *is* the most-real representation available — and as of 2026-07-05, "engine-native" is no longer just an inference from absent Lua source: disassembling `Desynced.exe` (UE5 Shipping build) with `radare2`/`r2ghidra` located the actual native `Tool.*` Lua-registration table and traced `GetClipboard`'s inner decode worker structurally against this module — the `"DS"` magic-byte check, the type-char position, and the base-31 length-header decode loop all matched with no discrepancies found (see memory for the full trace; stopped before the innermost checksum sub-function since the wire format was already cross-validated two other ways). The same table also revealed several native `Tool.*` functions this project doesn't use yet and hasn't documented elsewhere: `EncodeEntityStates`, `EncodeTrustLevels`, `GetTableDelta`/`ApplyTableDelta`, `TableToString`/`StringToTable`, `CompareStringNoCase`, `ContainsStringNoCase`, `MatchLocalizedRichText`, `GetDateStr`, `GetTimeDurationStr`, `ParseEntityStates`; the old tool's 0-based Python dict rendering was an artifact of JSON/Python convenience, not something intrinsic to the format, and it caused a real bug (`decode_dcs()`'s native return used `int` keys while hand-built/JSON dicts used `str` keys — every `dsc_codec.py decode <file>` CLI example elsewhere in this project only ever *looked* string-keyed because that was `json.dumps` doing the coercion for display, not the decoder itself). Building straight into Lua's own 1-based array convention removes that whole class of bug along with the representation layer that caused it. Still the same base62/zlib/msgpack-variant bytes on the wire — cross-validated against the retired tool's own output across all real `.dcs` files in this repo (exact structural match both directions) before it was deleted, and against the official [StageGames/DesyncedJavaScriptUtils](https://github.com/StageGames/DesyncedJavaScriptUtils) `dsconvert.js` reference implementation before that (two real bugs found in *their* encoder, documented in `dcs_wire.py`'s module docstring, deliberately not reproduced). It does **not** use a standard MessagePack library (repurposes type bytes `0xc1`/`0xc4`/`0xc5` for non-standard meanings) and has no instruction-metadata layer or disassembler — string ⟷ Lua-table transport only.
- `engine_stub.lua` / `lua_runtime.py` — the `lupa`-backed runtime. Key discovery, documented in both files: `Get`/`Set`/`GetNum` are **not** external engine globals as `behavior_format.md` originally assumed — they're local aliases *within* `instructions.lua` itself for real `InstGet`/`InstSet`/`InstGetNum`/`InstBeginBlock` functions also defined in that file, resolving arguments through a `GetStack`-based unified integer addressing scheme (`state.mem[]` for locals/literals, `comp.owner:GetRegister()` for frame registers). These are reused **verbatim, unmodified** — only the genuinely-external pieces are stubbed: the `Value` type's arithmetic (confirmed engine-side), a `comp`/`comp.owner`/`Tool.NewRegisterObject` stand-in, and `GetCachedBehaviorAsm` (needed for `jump`/`label`). `LupaEngine.decode_dcs`/`encode_dcs` wrap `dcs_wire.py` for the `Tool.GetClipboard`/`SetClipboard`-equivalent conversion (see above for why they're exposed as plainly-named methods rather than bound onto those exact Lua names — the real functions read/write the OS clipboard with no string argument, which isn't what we're simulating).
- `interpreter.py` — runs a behavior given as a genuine Lua table (1-based, the shape `dcs_wire.decode_dcs()` hands back and `compiler` produces directly — no Python dict anywhere in this module) by delegating each leaf instruction to the real Lua (via `lua_runtime.py`) while keeping the branch/block control-flow driver in Python (the same model `behavior_format.md`'s "Block-type instructions" section documents, validated against real in-game logs earlier in this project). Validated: the real `HexAt` sub-behavior (decoded straight from `HexIndexOf_test_1.dcs` via `dcs_wire.py`) run through this interpreter matches the known-good reference for all 91 in-domain `(R, T)` cases, with every arithmetic/branch/coordinate instruction executing genuine game Lua. `check_number`'s branch decision is now also genuinely delegated (resolve If Larger/If Smaller to raw asm targets ourselves, pass them straight through as the real func's own args, read `state.counter` back) rather than re-derived in Python — an earlier draft only delegated the value reads and re-did the comparison itself, confirmed behaviorally equivalent (including `REG_INFINITE`) but not literally "real," before being fixed to match `jump`'s already-fully-delegated approach. `sequence`/`for_number`'s block-stack driving is simulated in Python rather than reusing the real `InstBeginBlock` (confirmed present and callable in `data/instructions.lua`) — genuinely reusing it is still deferred, since it requires also replicating (or reusing) the real per-tick dispatcher's dead-end/`state.blocks` handling in `data/library.lua`, not yet examined. `for_number` and `last` (Break) are now both implemented on top of that same Python-simulated block stack (previously neither was handled at all — any compiled program using either op would hit the `unhandled op` fallback), following the convention above: the per-iteration advance/termination decision (including the documented `Step` auto-direction quirk and `REG_INFINITE` wraparound) is genuinely delegated to the real `data.instructions.for_number.next`/`.last`, with only the small, already-pinned-down initial-offset formula ported by hand (calling the real `BeginBlock` for this would require adopting the real dispatcher's own `state.counter` pre-seeding convention, a bigger change bundled with the `InstBeginBlock` reuse above). Fixing this also surfaced and fixed a real pre-existing gap: falling off the true end of the instruction array without an explicit `next: false` never consulted the block stack at all, silently ending the run instead of popping to an enclosing loop/sequence — the main dispatch loop now checks bounds itself and calls the same dead-end path every time, not just on an explicit `false`.
- `compiler/ast_compiler.py` — compiler using Python's own `ast` module to parse a subset of Python syntax directly into a genuine Lua table (takes a `lupa.LuaRuntime` to build it). Grew substantially from a first-cut (assignment, `+ - * //`, a single flat `if a > b / a < b: ... else: ...`) while compiling `hex_expansion_math.md`'s revised `HexIndexOf` (see that section below): arbitrarily nested arithmetic and unary negation now hoist sub-expressions into fresh temporaries instead of only accepting one flat `a op b`; `if`/`elif`/`else` also handles `==`/`!=`/`>=`/`<=` and a top-level `a and b` chain (still no `or`, no nested boolean expressions); `if cond: continue` (no `else`) inside a loop compiles to an explicit `false`-wired branch rather than real instructions — the same "not a match, try the next one" idiom this project's hand-authored behaviors already use, just now recognized as a distinct pattern from a normal `if`/`else` (its "which outcome dead-ends" wiring is the *complement* of the normal case's, a real bug caught and fixed during this work: the first draft wired the wrong outcome to `false` for every op except the `if`/`else` cases it happened to share code with); `for x in range(a, b):` / `break` compile to `for_number`/`last`; `jump(value)`/`label(value)` are recognized as bare statement-calls; `abs(x)`/`max(a, b, ...)` macro-expand into the check_number-guarded idioms `hex_expansion_math.md` documents (no native instruction for either) by synthesizing an equivalent `if`/assignment AST and recursing, rather than hand-emitting new instruction-building code; `a, b = separate_coordinate(x)` and `x = combine_coordinate(a, b)` call the real coordinate instructions directly. A real equality-handling bug was found and fixed early on: the `if a > b` form used to route the `a == b` case into the wrong branch (fixed by always compiling the Python `else` as the physically-next fallthrough, regardless of which comparison direction is being tested — see the instruction's own comment for why). Confirmed to round-trip through `dcs_wire.py`'s encode/decode cycle. Still no declared sub-behavior `parameters`/`pnames` (a compiled program's inputs/outputs are just plain named locals, seeded/read directly through `Interpreter.mem` in tests) — see `tests/test_hexindexof_compiled.py` for the current ceiling of what this compiles.

**Not done yet:** reusing the real `GetFactionBehaviorAsm` compiler from `data/library.lua` (this harness has its own simplified stand-in for now, `Interpreter`'s `Memory`/mem-slot allocation); reusing the real `InstBeginBlock` for `sequence`/`for_number` block-stack driving itself (both still Python-simulated, though `for_number`'s own per-iteration decision is now delegated to real Lua — see `interpreter.py` above); declared sub-behavior parameters/`call`; `or`/nested boolean expressions in `if`; coordinate/id/entity literals in source syntax. **A decompiler is now further along than "not started"** — as of 2026-07-06 this Python-like target itself was superseded as the intended decompiler output shape by `behavior_source_format.md`'s graph-native format instead (see "Design Docs" below and `scripts/render_examples.py`, which is a working prototype of the read/extraction half); this `ast_compiler.py` module still stands as the only *compile* (source → table) direction that exists, and reconciling it with the new target format is itself a not-yet-started piece of work, not just "decompiler not started" as this line used to say.

## Design Docs Built on This Data

`instructions_index.md` (workspace root) is an auto-generated reference of every entry in `data.instructions` (`data/instructions.lua`) — visual-editor name, op id, and argument list (in/out/exec, in declaration order, with filter type tags). Regenerate by re-parsing that file if it changes. `behavior_format.md` (workspace root) is the companion reverse-engineered spec for the *wire* format a behavior decodes into/from — register/slot addressing, branch and fall-through resolution, `make_asm`/hidden-literal (`c`) fields, and variable-length-arg instructions — cross-checked line-for-line against `observer.dcs`. It predates `desynced_toolkit.dcs_wire` and was originally written against the retired `dsc_codec.py`'s 0-based Python dict rendering; its description of the *actual wire format* (register addressing, branch resolution, hidden literals, etc.) is still accurate, but where it talks about "dsc_codec.py" or 0-based dict keys as if that's how the data is held in memory, that's the old tool's rendering choice, not the format itself — `dcs_wire.py` now hands back the same information as genuine 1-based Lua tables instead. Together with `instructions_index.md`, these are the missing layer between the wire bytes (transport only, no instruction semantics) and hand-authoring a new behavior; read both before writing or editing any `.dcs` content by hand, or before answering *any* question about register/value semantics, branch/jump encoding, or behavior control flow — not just when actively editing a `.dcs` (see "Non-obvious engine/format gotchas" below for the index of specific traps this file documents).

`behavior_source_format.md` (workspace root) is a layer above `behavior_format.md`, not a replacement for it: a specification for a graph-native textual read/edit representation of behaviors (nodes with explicit branch edges, not tree-structured pseudocode), adopted 2026-07-06 after empirical analysis of 319 real user-created behaviors (see project memory) found tree-structured syntax a poor fit for a large share of real cases — `jump`/`label` computed dispatch alone (20.7% of real behaviors) is indirect goto by construction and cannot be expressed as `if`/`else`/`for`. Read it before implementing or reasoning about any decompiler/compiler work in `desynced_toolkit` — it supersedes `compiler/ast_compiler.py`'s Python-like target as the intended direction, though that code hasn't been replaced yet (see its own "Status" section for exactly what's built vs. still spec-only).

`combat_squad_spec.md` (workspace root) is a hand-written behavior design spec (squad AI roles: Beacon/Scout/Gunner/Support) built from real instructions in `data/instructions.lua` and components in `data/components.lua` — a consumer of this extract, not part of the extract itself. `observer.dcs` is a companion example behavior clipboard string it references, decodable via `desynced_toolkit.dcs_wire` (or, historically, the now-retired `dsc_codec.py`). `beacon.dcs` is a hand-authored implementation of the spec's §5.1 Beacon role (built using `instructions_index.md` + `behavior_format.md`, round-trip-verified against the codec) — target selection is implemented via reservoir sampling rather than the spec's literal shortlist-array pseudocode (a proven equivalent that avoids Memory-array instructions and an unconfirmed `REG_INFINITE` literal), and exact-dedup of duplicate reports is intentionally omitted (no native set/contains instruction exists; the spec's own §7 already treats near-tie imprecision as acceptable). Not yet tested in-game. (`observer.dcs`/`beacon.dcs`/`beacon2.dcs` all live under `tests/data/` now — see the note at the end of this section.)

`hex_expansion_math.md` (workspace root) is a hand-worked-out coordinate-math spec for a hexagonal-spiral power-pole expansion behavior (`HexAt(origin, R, T) -> coord` and its inverse `HexIndexOf`), built against the integer-only constraints of the Math instructions. `HexAt` is implemented and validated in-game: `hexat_test.dcs` (`tests/data/`) is a self-contained `.dcs` bundling both a reusable `HexAt(R, T, Origin, d_half) -> Result` sub-behavior (via the top-level `dependencies` mechanism documented in `behavior_format.md`) and a test harness that calls it for every `R=0..5, T=0..6R-1` and `debug_print`s the result; `hexat_test_log.txt` (workspace root — a captured log, not a `.dcs`, so it stayed put) is the captured log from that run. Cross-checked by hand against `hex_expansion_math.md`'s formulas for all 92 logged `(R, T)` cases with zero mismatches. ## Non-obvious engine/format gotchas — check before answering

These are confirmed-the-hard-way facts — each one was either misread from Lua source alone and corrected against real in-game behavior, or is a sharp edge easy to assume the "obvious" wrong thing about. `behavior_format.md` has the full detail, worked examples, and confirmation method for each; this index exists so the fact of a gotcha's existence surfaces even when the current question doesn't look like ".dcs editing" (e.g. a plain conceptual question about how register values work). Check the cited section before asserting any of these from memory, from the Lua source alone, or from general assumption:

- **Register/value composite semantics** — `coord`/`entity`/`id` are mutually exclusive on a value (setting one clears the other two); `num` is separate and coexists with any of them. `add`/`sub` combine them non-uniformly by operand shape (coordinate⊕coordinate adds both parts including `num`; coordinate⊕bare-number broadcasts the bare number onto both coordinate components and leaves the coordinate side's `num` untouched; entity/item⊕bare-number passes the entity through and adds `num` normally). `behavior_format.md` § "Composite values and the `num` field" (~line 137).
- **Coordinate literal shape** — must be `{"coord": {"x": x, "y": y}}` (keyed object), never a `[x, y]` array — the array form round-trips fine through the codec but loads in-game as silently corrupted. `behavior_format.md` ~line 124.
- **`exec`/`next` jump targets are raw 1-based Lua positions**, not the 0-based dict keys used elsewhere in decoded output — subtract 1 to get the dict key (add 1 when hand-authoring). `behavior_format.md` § "Off-by-one" (~line 248).
- **`exit` genuinely halts the behavior** — nothing runs again until something else explicitly restarts it, despite `state.counter = 1; return true` reading from source alone like "yield and resume next tick." `restart` unconditionally jumps to Program Start (unwinding the *entire* subroutine call stack in one step if called from inside a `call`ed sub-behavior, not just returning one level) but does **not** yield — it can still spin forever under `unlock` with no other yield point. `behavior_format.md` § "Stopping a behavior" (~line 321).
- **`next: false` inside a block (loop/`sequence`) pops to the *enclosing* block's own next-step logic**, not Program Start — only true top-level fall-off does that. `behavior_format.md` § "Block-type instructions" (~line 372).
- **`check_number`'s "Equal" outcome isn't a numbered arg slot** — it's carried by the instruction's ordinary top-level `next`/fallthrough. To test equality, wire both `If Larger` and `If Smaller` to the same "not equal" target and let fallthrough be "equal". `behavior_format.md` ~line 429.
- **An omitted `exec` arg never consults the top-level `next`** — `If Larger`, `If Smaller`, and `next` (Equal) are three independent slots, each with its own fallthrough-to-next-instruction default; setting `next` does not also cover an omitted `If Smaller`/`If Larger`. `behavior_format.md` ~line 445.
- **`for_number`'s omitted `Step` auto-detects direction from `From`/`To`**, including flipping sign at runtime for a bound expression that can invert (e.g. `6*R - 1` at `R=0`) — this silently reverses direction and runs at least once, rather than running zero times the way a fixed-step loop would. `behavior_format.md` ~line 414.
- **`call`'s `sub` field** is a string (saved-library id), *or* a small positive int (1-based index into the top-level `dependencies` array for an embedded sub-behavior), *or* `-1` (recursive self-call sentinel, distinct from the legacy runtime's `0`) — never assume it's always a library id. `behavior_format.md` ~line 506.
